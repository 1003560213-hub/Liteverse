# Liteverse Curator

Prepare deterministically, review source-pinned evidence in bounded batches, then publish one complete staged revision. Never edit `Graph/current.json`.

## Workflow

1. Resolve the support directory from `--support-dir`, `LITEVERSE_SUPPORT_DIR`, or `~/Library/Application Support/Liteverse`.
2. Run `scripts/list-queue.mjs --json`. Record each literature or annotation ID and revision before reading files. Refuse to finalize an item whose revision changed.
3. Inspect each queue item's optional `preparation` record and `source`. Treat absent `source.storageMode` as legacy `managed`. For `linked`, including `source.provenance.catalog: zotero`, require the normalized absolute PDF path and root/relative provenance, re-hash the external file, and never copy it into `Library/PDFs` or an artifact revision. Do not write to Zotero or reinterpret unsupported Zotero attachment modes. When preparation is `ready`, verify the local-job manifest and reuse its hash-pinned page-marked full text, card skeleton, and routing-only Review Packet. When absent, use `scripts/materialize-paper.py` only for legacy managed sources. Reject queued, needs-attention, stale, missing, or hash-mismatched output.
4. Read `references/review-batch-contract.md`. Run `scripts/build-review-batch.mjs` to create a deterministic 3–5-paper batch, bounded by candidate characters and balanced across scientific evidence kinds; use `--allow-partial` only to drain a final one- or two-paper queue. Do not replace this with sequential whole-PDF reading. Review Packet v2 anchors identify high-value original pages, but never constitute evidence. Reopen each cited original PDF page and return exactly one `accept`, `qualify`, or `reject` decision per candidate. Keep assumptions, negation, units, conventions, and applicability limits explicit.
5. Run `scripts/apply-review-batch.mjs` with the hash-pinned decision JSON. Decide only the scientific candidates; navigation-only section, figure, table, and citation anchors require no response. When many candidates are not adopted, use the explicit per-paper `defaultUnspecifiedDecision: "reject"` and one reason instead of generating repetitive rejection prose. Let the script assign Evidence IDs and compose the review result; do not hand-edit IDs or reformat the Markdown mechanically. The default remains `card_draft`. Request `evidence_verified` only with the exact original-page-review attestation required by the contract. Resume the active checkpoint after interruption instead of rebuilding or rereading completed papers. Treat `needs_ocr` as needs attention and never infer a summary from title or filename.
6. Continue building and applying batches until every paper in the same preparation wave has a committed result. Then run `scripts/adopt-review-results.mjs` once with every result manifest as a repeated `--result`. Do not advance `Knowledge/papers.json` after the first batch while later batches still pin the earlier catalog. One multi-result adoption revalidates strict duplicate keys and commits the complete wave without re-extracting PDFs. Omit `--assignments` to keep papers provisional in `liteverse-staging`; use a hash-pinned assignments file only for scientifically reviewed existing-region assignments. Adoption emits a working snapshot and library-items file, but does not publish immutable truth or stage Refresh.
7. Read `references/taxonomy.md`. If a valid selected macro taxonomy already exists and the user did not request repartition, run `scripts/screen-incremental-classification.mjs` for only the new papers, review its routing-only top candidates, and scientifically confirm one primary plus at most one secondary region. Do not regenerate three full-corpus options for ordinary additions. Propose a new taxonomy only for first initialization, explicit repartition, or a batch of at least four low-fit papers whose provisional within-cluster consistency reaches 70.
8. When a taxonomy proposal is required, search the complete verified corpus and design exactly three materially distinct options with 1–10 macro regions, complete primary assignments, optional single secondary assignments, evidence IDs, strategy, rationale, and structured strengths/limitations. Run `scripts/compose-partition-options.mjs`, then `scripts/propose-partitions.mjs`. Present all three and stop unless the user explicitly chooses or delegates selection.
9. After an explicit or delegated choice, run `scripts/apply-partition-choice.mjs` with `--confirmed-by-user`, a confirmation note, and ISO decision time. For delegation, record `delegated choice` plus the scientific basis. The script rechecks revisions and fingerprints, appends the decision, assigns unused-first nebula assets, and regenerates deterministic 3D positions. Never infer a choice from silence. Use `--rebuild-selected` only for the exact existing choice.
10. Read `references/relation-rubric.md`. Reuse each batch paper's hash-pinned `relationShortlist` instead of repeating a full-library search; supplement it only with explicit citation matches or same-batch candidates. Inspect no more than 24 routing candidates per new paper unless the user explicitly requests a full audit. Read located evidence from both original papers, then pass the scoring JSON to `scripts/score-connection.mjs`. A routing rank is never relationship strength or confidence.
11. Treat handwritten notes as user observations or questions until checked against the original PDF. Update cards and relationships only with retained annotation ID and revision provenance.
12. Run `scripts/finalize-curated-snapshot.py` against the adoption's unstaged working snapshot. It re-hashes managed or linked PDFs; a missing or changed linked source fails closed. For verified/needs-attention cards it requires original-source locators, valid `[E#]` references on every scientific bullet, and positive page markers.
13. Run `scripts/generate-claims.mjs --paper <new-id> ... --snapshot <same-working-snapshot>` to create stable sidecars and immutable hash-pinned revisions. Reuse unchanged artifact revisions across projects, layout changes, taxonomy screening, and repeated batches. Every adopted paper must have an integrity pin in the working snapshot before staging.
14. Merge reviewed relation outputs with `scripts/merge-relation-review.mjs --support-dir <path>`. Use `--require-all` only for a deliberate full legacy re-score. Finalization and merge refuse current, staged, and history graphs.
15. Validate and publish the complete staged revision with `scripts/stage-refresh.mjs --library-items <adoption-library-items.json>`. It materializes the nebula→galaxy→paper routing hierarchy, re-hashes linked PDFs, writes `Graph/staged/<refresh-id>/` and `Graph/pending-update.json`, and never promotes current.
16. Mark exactly one annotation with `scripts/mark-annotation.mjs --id <id> --revision <n> --refresh-id <id> --derived-file <path>`. Never bulk-mark annotations.

## Maintenance-only localization and relayout

- Use `scripts/prepare-layout-localization.mjs` only when the user explicitly asks to retain the selected taxonomy while translating every graph-visible title, category, paper summary/project role/tag, and relation label, and/or recomputing all existing nebula centers. Supply an exact-ID English translation map, `--confirmed-by-user`, an English confirmation note, and an ISO decision time.
- Treat this as a new maintenance revision, not a partition choice. Preserve category, paper, and relation IDs; primary/secondary assignments; evidence; relationship scores; artifact pins; nebula assets; and Usage. Write an unstaged Planning snapshot, validate it, then publish it with `scripts/stage-refresh.mjs`.
- Never reuse a stale partition proposal to perform maintenance. Promotion still belongs exclusively to the App commit bridge.

## Non-negotiable rules

- Do not seed a fixed taxonomy or silently continue the historical four-region layout. Every partition decision starts with exactly three corpus-derived options and ends with one explicit user selection.
- Treat an explicit request for Codex to recommend and choose as delegated selection authority, not as silence. Keep `--confirmed-by-user`, and make `confirmationNote` state `delegated choice` plus the scientific recommendation basis. Without direct selection or this delegation, wait.
- Rewrite only recognized machine-generated region annotations in `projectRole`: the documented leading prefixes and sentence-boundary classification tails. Preserve the scientific-use body and ordinary academic uses of “classification.” Normalize any recognized legacy marker to the canonical English annotation.
- Do not silently promote `needs_attention` evidence. The compact composer may use it only when both paper and claim have that state and the claim retains evidence; mark every resulting assignment provisional and expose it in metadata.
- Do not create a narrow region for an isolated paper. Every newly created macro region requires at least four primary papers and at least 70% within-cluster consistency. For an automatic incremental region, every member must also fit all existing regions below 60%; for an explicit full-corpus repartition, record old-region scores transparently as 0–100 diagnostics but do not use the 60% cutoff to veto a competing taxonomy. Keep smaller groups in `liteverse-staging` as provisional; it is not a macro region.
- Do not let proposal or selection scripts write `Graph/current.json`, `Graph/staged/`, `Graph/pending-update.json`, queues, or Usage. The selected output remains unstaged until `stage-refresh.mjs` passes independently.
- Do not let localization, relayout, or local preparation change scientific identity, classification assignments, evidence, relationship scores, artifact pins, nebula assets, Usage, or immutable partition history.
- Do not establish a formal relationship without located evidence from both original papers.
- Do not treat Review Packet text, candidate scores, FTS/BM25 ranks, Zotero metadata, or explicit citation matches as verified scientific claims. They are routing caches only.
- Do not read every paper or every pair when a source-pinned Review Batch and Top-K shortlist exist. Reopen only the original pages needed to accept, qualify, or reject candidates, expanding to more pages when scientific ambiguity requires it.
- Do not regenerate three partition options for ordinary additions to a valid selected taxonomy. Use incremental screening until the initialization, explicit-repartition, or stable-novel-cluster gate is met.
- Do not translate legacy `verified` flags into `evidence_verified`. Mechanical extraction produces `card_draft` or `needs_ocr`; evidence verification requires a later original-source review.
- Preserve an old `confidence` as `legacyConfidence`; never reinterpret it as the new relationship strength.
- Do not modify `Usage/events.jsonl`, `Usage/counts.json`, `useCount`, or any visual temperature.
- Do not automatically download or add suggested literature outside the user's library.
- Do not make a shared paper wait for a fabricated project role. Leave it explicitly unassigned during ingestion; record a project-specific role through Research Memory only when that project adopts the paper.
- Do not promote staged data to `Graph/current.json`; only the App commit bridge may do that.
- Never use a finalization or relation-merge script to rewrite `Graph/current.json`, an already staged refresh, or graph history, even when those paths are outside the configured support directory.
- Do not overwrite a mismatched managed PDF or backup. Stop on hash conflicts and resume through the migration manifest after the conflict is resolved.
- Do not copy, relocate, repair, or silently repoint a linked PDF. Require `realpath(root) == linkedRootPath` and `realpath(pdf) == pdfPath`, require every component below the selected root to be non-symlinked, verify the PDF remains inside the selected root, and preserve its absolute reference and pinned SHA-256. When it is missing, changed, or unsafe, report `needs_attention` and require the user to restore or explicitly relink it.
- Do not include linked original PDFs in a Liteverse backup. Back up only their reference, hash, Markdown, claims, and graph provenance; after restore, use `liteverse doctor` to report unresolved links.
- Do not edit queue JSON by hand. Use `list-queue.mjs`, then preserve the locked revision through materialization, staging, and single-annotation marking.
- Treat App-generated review packets and candidate scores only as rebuildable routing aids. They may reduce the pages and pairs Codex must inspect, but they never constitute scientific evidence or a formal relationship.
- Treat galaxies as deterministic visual and retrieval routing inside a macro region, never as a second scientific taxonomy. Do not hand-author galaxy IDs, store a new galaxy relationship score, or modify Usage while building the hierarchy.
